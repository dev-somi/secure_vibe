# Next.js Framework Development Rules

이 문서는 프로젝트의 일관성과 유지보수성을 높이기 위해 Next.js 개발 과정에서 지켜야 할 핵심 안티그래비티 룰(Antigravity Rules)을 정의합니다.

## 1. 코드 품질 및 파일 구조 원칙

*   **알아보기 쉽고 간결한 코드 작성하기:** 다른 개발자가 코드를 읽을 때 직관적으로 이해할 수 있도록 명확한 변수명과 함수명을 사용합니다. 불필요하게 복잡하게 꼬아놓은 추상화나 과도하게 중첩된 삼항 연산자를 피하세요.
*   **관심사 분리와 작은 파일 지향:** 하나의 파일이 너무 커지거나(예: 300줄 이상) 단일 컴포넌트가 여러 역할을 수행하고 있다면 지체 없이 여러 파일과 하위 컴포넌트로 나눕니다. 파일 하나당 단일 책임 원칙(SRP)을 지키는 것을 목적으로 합니다.
*   **UI 로직과 비즈니스 로직 분리:** 복잡한 비즈니스 로직을 리액트 컴포넌트 내부에 두지 마십시오. 클린 아키텍처 구조에 맞게 `usecases`, `services` 혹은 커스텀 훅으로 분리하여 컴포넌트를 최대한 얇고 '멍청하게(Dumb)' 만드세요.

## 2. 서버 컴포넌트(Server Components) 최우선 지향 (핵심)

*   **Server Component First:** App Router를 사용할 경우 기본적으로 모든 컴포넌트는 서버 컴포넌트로 작성하는 것을 원칙으로 합니다.
*   **명시적인 클라이언트 컴포넌트 선언:** 상태 관리(`useState`), 생명주기 관리(`useEffect`), 브라우저 전용 API(`window`), 또는 사용자 상호작용(`onClick` 등)이 반드시 필요한 말단 노드 UI 컴포넌트에서만 파일 최상단에 `"use client"` 지시어를 사용하세요.
*   **서버/클라이언트 교차 사용 주의:** 서버 컴포넌트로 클라이언트 컴포넌트를 감쌀 수는 있지만, 클라이언트 컴포넌트 내부에 직접적으로 무거운 서버 컴포넌트를 임포트하여 렌더링하지 않도록 주의합니다(children prop 형태로 넘기세요).

## 3. 데이터 패칭(Data Fetching) 전략

*   **서버사이드 데이터 페칭 선호:** 데이터를 가져오는 작업은 무조건 최상단의 서버 컴포넌트에서 `async/await`와 네이티브 `fetch` API를 활용하여 수행합니다.
*   **데이터 뮤테이션:** 클라이언트 측에서 폼 제출 등의 이벤트로 데이터를 수정할 시, 가급적 Next.js의 **Server Actions**를 활용하여 API 라우트를 별도로 구축하는 리소스를 줄이고 타입 안전성을 확보합니다.

## 4. Next.js 내장 최적화 기능 적극 활용

*   **Image 최적화:** 기본 `<img>` 태그 사용을 금지하며, 반드시 `next/image`의 `<Image />` 컴포넌트를 사용합니다. 이는 CLS(Cumulative Layout Shift)를 방지하고 최신 포맷을 지원합니다.
*   **Navigating 최적화:** 내부 라우팅 시 기본 `<a>` 태그 대신 `next/link`의 `<Link href="...">`를 사용하여 링크 프리패칭과 매끄러운 클라이언트 사이드 전환 이점을 극대화합니다.
*   **폰트 최적화:** 외부 폰트(구글 폰트 등) 사용 시 레이아웃 시프트를 줄이기 위해 `next/font/google`과 같은 내장 폰트 모듈을 활용합니다.

## 5. 라우팅 및 폴더 코로케이션(Colocation)

*   **라우팅 요소 외 파일 혼재 관리:** Next.js App Router는 `page.tsx`, `layout.tsx`, `route.ts` 등의 예약 파일만 라우팅에 영향을 줍니다. 특정 페이지(Route)에 완전히 종속된 특정 하위 컴포넌트나 상태 파일은 해당 디렉터리 내부에 공존(Colocate)시키는 것을 권장합니다.
*   하지만 2곳 이상에서 재사용될 여지가 생기는 순간, 즉시 `src/components/` 혹은 독립된 모듈 폴더로 위치를 이동시킵니다.

## 6. 엄격한 타입스크립트(TypeScript) 환경 유지

*   **`any` 타입의 엄격한 금지:** 모든 컴포넌트, Props, 파라미터 및 반환값에 명시적인 타입을 부여합니다. 불가피한 경우라면 `unknown`을 활용하고 타입 가드를 거치세요.
*   **안전한 환경 변수 처리:** `process.env` 값을 직접 호명하기보다 Zod 등 유효성 검사 라이브러리를 활용해 환경변수의 타입을 검증하는 레이어를 두는 것을 권장합니다.
